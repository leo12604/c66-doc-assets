## 0. 總覽 Executive Summary

> 一頁讀懂這個產品：是什麼、給誰、怎麼運作、最近半年發生什麼。

**這是什麼**  
本產品是嵌入在 **BingoPlus / SharePlus** 主站內的**社群模組**——讓玩家在玩遊戲之餘，能瀏覽內容、發表動態、互動社交、領取任務獎勵，並參與營銷活動。

**給誰用**  
主站既有玩家，特別是有社交與分享傾向、願意產生 UGC（用戶生成內容）的高頻使用者。

**核心商業循環**

```
拉新 ──► 留存 ──► 活躍 ──► 變現 ──► 召回 ──► 拉新（循環）
 │        │        │        │        │
裂變    任務     互動     導流回    推播
活動    領獎     UGC      主站     通知
```

**半年下來累積的四大成果軸**

| # | 成果軸 | 主要已上線能力 | 對應商業價值 |
|---|---|---|---|
| ① | **沉浸式內容體驗** | 雙欄 / 單欄兩種瀑布流；沉浸式短影片；合集 | 瀏覽時長、內容黏著 |
| ② | **每日任務體系** | 每日任務、連續任務、領獎體驗、任務風控 | DAU、留存 |
| ③ | **社群基礎建設** | 通知中心兩層架構（頻道 / 訊息）、全場景站內訊息、追蹤與新粉絲提示 | 召回率、社交網絡密度 |
| ④ | **營銷活動矩陣** | 金剛區（TAROT / 每日任務 / 遊戲測評 / 遊戲偏好分析 / 排行榜 5 個常駐入口）、九宮格裂變、品牌故事、世界盃、Carlo | 拉新、季節性運營、IP 合作 |

## 1. 產品定位

### 1.1 產品形態

本社群模組不是一個獨立 App，而是**嵌入在 BingoPlus / SharePlus 主站內**的子模組。玩家透過主站的某個入口（如底部 Tab、活動 Banner、推播訊息）進入社群，使用者身份由主站直接帶入，**不需要重新註冊或登入**。

```mermaid
flowchart LR
    A[BingoPlus<br/>主站玩家] --> B[底部 Tab / Banner / 推播]
    B --> C[進入社群模組]
    C --> D[瀏覽內容]
    C --> E[完成任務]
    C --> F[發表動態]
    D --> G[回主站玩遊戲]
    E --> G
    F --> G
```

對玩家而言，社群與主站是「同一個體驗的不同面向」：主站打遊戲、社群聊遊戲，兩者互相導流。

### 1.2 目標受眾

| 受眾類型 | 在社群中的行為 | 對商業價值的貢獻 |
|---|---|---|
| **高頻遊戲玩家** | 看別人的遊戲分享、追蹤其他玩家、領每日任務獎勵 | 留存、活躍 |
| **內容創作者（PUGC）** | 發遊戲心得、贏分截圖、影片合集 | 內容供給、社群活力 |
| **對社群陌生的主站玩家** | 透過裂變活動連結或主站入口進到社群，看排行榜、追蹤熱門帳號（玩家身份仍由主站帶入） | 拉新轉化（社群內活躍） |
| **沉睡用戶** | 透過通知 / 推播被召回 | 召回率 |

### 1.3 商業價值定位

社群模組在主站生態中扮演三個角色：

1. **黏著層**：用內容與社交把玩家留在 App 內，延長 session 時長
2. **拉新層**：透過分享、裂變、邀請機制把現有玩家的社交網絡轉為新用戶
3. **召回層**：透過即時訊息與通知把流失玩家帶回 App

### 2.1 進場：玩家從哪來、第一眼看到什麼

**商業價值**：拉新 / 跨模組導流

**進入路徑**  
玩家從主站三種入口進入社群：
- **底部 Tab**：主站底部固定的「社群」分頁（高頻入口）
- **活動 Banner**：主站首頁活動位推廣特定社群內容（短期運營入口）
- **推播 / 站內訊息**：玩家收到推播或站內訊息點擊進入（召回入口）

**進入後第一眼**  
玩家落地在**首頁**，預設看到：
- **金剛區（首頁頂部活動入口 Banner）**：常駐運營活動入口，可橫向滑動。目前包含的入口：
  - **塔羅牌（TAROT）**：新手趣味互動
  - **每日任務（DAILY_TASK）**：留存核心引擎
  - **遊戲測評（GAME_REVIEW）**：玩家對遊戲的評價內容
  - **遊戲偏好分析（GAME_DNA）**：建立個人遊戲偏好標籤
  - **排行榜（RANKING）**：玩家排名與競爭運營
  - 部分入口會帶 **Hot / New** 等狀態標籤
- **頂部分類列**：不同遊戲類型 / 主題分頁（精選 / 追蹤 / 遊戲類型）
- **主體區瀑布流推薦內容**（混合圖文、影片、合集）
- 賽事或主題活動期間，頂部會替換為對應主題背景（例：世界盃情報背景僅在賽事活動期間展示，非長駐 UI）

**對 PM 的設計重點**  
玩家進場「零摩擦」是社群留人的關鍵。沒有登入頁、沒有歡迎彈窗（除非有特定運營活動需要），玩家來了就直接看到內容。

### 2.2 內容消費：feed → 沉浸式 → 詳情頁

**商業價值**：留存 / 黏著 / 時長

這是整個社群模組**最核心的場景**，玩家絕大多數的瀏覽時間都花在這個流程上。

```mermaid
flowchart TD
    A[首頁瀑布流] --> B{點擊內容卡片}
    B -- 影片 --> C[沉浸式全屏播放]
    B -- 圖文 --> D[詳情頁]
    B -- 合集卡片 --> E[合集頁]
    C --> F[上下滑切換內容]
    D --> G[滑到底瀏覽更多]
    E --> H[點合集內單篇]
    H --> C
    H --> D
```

**首頁瀑布流（Feed）**  
- 不等高卡片左右錯落呈現（瀑布流佈局）
- 內容混合：圖文帖子、短影片、合集封面、贏分分享、品牌故事卡片
- 滑到底自動載入下一批
- 支援頂部分類切換（如：精選 / 遊戲類型 A / 遊戲類型 B / 追蹤）

**沉浸式（Immersion）**  
- 點影片內容後進入全屏播放
- 上下滑切換到下一個影片（類 TikTok 體驗）
- 螢幕左下顯示作者頭像、暱稱、Follow 按鈕
- 螢幕右下橫向排列互動按鈕（留言、分享、讚）
- 底部固定留言輸入框，可直接留言

**詳情頁**  
- 點圖文內容進入完整閱讀
- 含完整圖文、互動區、評論區、相關推薦

**合集頁**  
- 創作者打包多篇內容形成的集合
- 玩家可逐篇瀏覽，享受連續性內容體驗

**瀑布流呈現方式（雙瀑布流 / 單瀑布流）**

| 雙瀑布流（兩欄不等高錦拼） | 單瀑布流（單欄全寬卷動） |
|---|---|
| <img src="screenshots/01-home-feed.png" alt="雙瀑布流首頁" width="180"> | <img src="screenshots/18-home-single-column.png" alt="單瀑布流" width="180"> |

玩家可由首頁右上角的視圖切換按鈕在兩種瀑布流之間切換。

**多圖貼文呈現（詳情頁）**：一篇 post 含多張圖片時，玩家可在詳情頁橫向 / 縱向切換瀏覽。

### 2.3 互動與社交：讚 / 評論 / 分享 / 收藏 / 追蹤

**商業價值**：UGC 增量 / 社交網絡密度 / 病毒散播

玩家瀏覽內容時，五種互動行為決定社群活力：

| 互動 | 玩家動機 | 商業意義 |
|---|---|---|
| **點讚** | 快速表達喜歡 | 內容熱度信號（影響推薦） |
| **評論** | 深度表達意見、參與討論 | 內容深度互動、UGC 增量 |
| **分享** | 把內容轉發給朋友 / 外站 | 病毒散播、拉新入口 |
| **收藏** | 留存稍後再看 | 用戶資產累積 |
| **追蹤** | 持續關注某創作者 | 創作者粉絲化、回訪驅動 |

**關鍵設計**  
- 點讚有**動畫回饋**（愛心彈出），強化即時滿足感
- 評論可在帖子內展開回應與互動
- 分享支援站內分享、站外分享（以短網址呈現）、贏分截圖三種模式
- 玩家追蹤其他人後：自己的「追蹤」分頁會出現該創作者新動態；對方則會在「新粉絲」分頁看到自己被誰追蹤

**互動行為的入口**多在瀑布流卡片下方（讚 / 評論計數）、沉浸式右下按鈕、詳情頁的互動區塊。

### 2.5 任務與獎勵：每日任務 → 領獎 → 財富中心

**商業價值**：DAU / 留存

這是**留存的核心引擎**。透過任務目標 + 獎勵的循環，把「來看內容」的玩家轉為「每天回來簽到」的玩家。

**任務狀態流轉**

```mermaid
stateDiagram-v2
    direction LR
    [*] --> 未開始
    未開始 --> 進行中: 玩家開始做任務
    進行中 --> 可領取: 達成目標
    可領取 --> 已領取: 玩家點領獎
    已領取 --> [*]
```

**任務類型**  
- **每日任務**：每天重置，例：瀏覽 N 個內容、點讚 N 次、發 1 則動態
- **連續任務**：連續打卡的長線任務
- **特定主題任務**：搭配運營活動的限時任務

**領獎體驗**  
- 任務狀態變化時**主動彈出提示**（達成目標、可領獎時系統會即時告知，避免玩家錯過獎勵）
- 任務列表中已領取與可領取的任務以不同按鈕樣式區分，引導玩家完成下一步動作
- 領取後跳轉**財富中心**查看獎勵餘額

**每日任務畫面**：頂部獎金池、任務列表、依任務狀態顯示「可領獎」或「前往」按鈕、時間軸與進度

### 2.6 通知與召回：通知中心、全場景訊息、紅點

**商業價值**：召回率 / 訊息觸達 / 喚醒沉睡

**通知中心結構**：頂層有兩個 Tab（**頻道** / **訊息**），「訊息」Tab 內再分 3 個入口。

| 層級 | 名稱 | 內容 | 商業價值 |
|---|---|---|---|
| 頂層 Tab | **頻道（Channel）** | 玩家關注的遊戲 / 主題頻道列表，含更新提示 | 創作者粉絲化、主題運營 |
| 頂層 Tab | **訊息（Chat）** | 與其他玩家 / 自己內容相關的 3 類訊息入口（見下方） | — |
| 訊息 Tab 內 | **新粉絲** | 誰追蹤了你（含「回追蹤」按鈕） | 創作者激勵 |
| 訊息 Tab 內 | **互動通知** | 你發的內容收到的讚 / 評論等內容互動回饋 | 內容互動回饋 |
| 訊息 Tab 內 | **系統通知** | 官方公告、版本更新、帳號審核結果（如發文 / 頭像審核） | 用戶教育 |

**紅點機制**  
- 通知中心頂層 Tab 與「訊息」Tab 內各入口皆有未讀紅點
- 點進去後對應分頁紅點清除
- 未讀數即時更新

**全場景訊息與通知**  
- 玩家在社群模組的**任意頁面**都能即時收到「站內訊息」（不限於通知頁）
- 覆蓋形式：站內彈出提示、訊息列表更新、紅點刷新
- **範圍邊界**：以上指「站內訊息」，玩家需在 App 前台時才會收到；OS 層的「系統推播」由主站統一管理（頻控、配額等規則見 5.4）

## 5. 產品邊界與限制

> 不講技術邊界，講「PM 設計功能時必須知道」的產品 / 商業邊界。

### 5.1 身份與權限：依附主站

- **無獨立帳號**：玩家身份由主站帶入，社群內不可獨立註冊 / 登入
- **登入流程零摩擦**：玩家不會在社群內遇到任何登入頁
- **影響設計**：拉新動作的最終承接者是主站；社群活動的拉新動能最終會匯入主站開戶率

### 5.2 內容類型

目前產品支援的內容類型：
- ✅ 圖文帖子
- ✅ 短影片
- ✅ 合集（多篇內容打包）
- ✅ 贏分截圖分享
- ❌ 長影片（不支援）
- ❌ 純文字長文（不支援）

### 5.3 上線節奏（灰度 → 正式）

新功能上線會走「灰度 → 正式」兩階段：
- **灰度期**：先對小比例用戶開放，觀察數據與體驗
- **正式期**：通過灰度評估後全量發佈

對 PM 的意義：**新功能規劃時要預留灰度觀測期**，不能假設一上線即觸達全量用戶。每月會有一個發佈批次，依此節奏對齊運營排期。

### 5.4 跨模組依賴

社群模組對主站有以下依賴（PM 規劃功能時需確認跨團隊協作）：
- **用戶身份系統**：主站維護
- **獎勵 / 財富帳本**：主站維護（社群只展示）
- **遊戲列表 / 詳情**：主站維護
- **支付 / 兌換**：主站維護
- **OS 層系統推播通道**：主站統一管理（社群只發站內訊息；系統推播的頻控、配額、退訂規則由主站決定）
